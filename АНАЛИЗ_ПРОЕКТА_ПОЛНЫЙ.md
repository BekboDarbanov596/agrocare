# ЧАСТЬ 1. ТОЧКА ВХОДА — index.php

**Файл:** `index.php` (корень проекта)

**Что выполняется по шагам:**

1. **Строки 4–6.** Включение отображения ошибок PHP:
   - `error_reporting(E_ALL)` — все ошибки;
   - `ini_set('display_errors', '1')` и `display_startup_errors`, `'1'` — показ на экране (для отладки).

2. **Строки 9–11.** Запуск сессии, если она ещё не начата:
   - `if (session_status() === PHP_SESSION_NONE) { session_start(); }`.

3. **Строка 13.** Подключение автозагрузки Composer: `require_once __DIR__ . '/vendor/autoload.php'` — подгружаются все классы из `vendor/` и пространства имён по PSR.

4. **Строка 14.** Подключение глобальных хелперов: `require_once __DIR__ . '/app/Core/helpers.php'` — там определена функция `view()`.

5. **Строка 16.** Импорт класса приложения: `use App\Core\Application;`.

6. **Строки 19–21.** Загрузка переменных окружения:
   - `Dotenv\Dotenv::createImmutable(__DIR__)` — объект для папки с `.env`;
   - `$dotenv->load()` — заполняет `$_ENV` из `.env` (DB_*, HUGGINGFACE_API_KEY и т.д.).

7. **Строки 24–26.** Создание и запуск приложения:
   - `$app = new Application();` — внутри создаются Request, Router, Database и подключается `routes.php`;
   - `$app->run();` — вызывается `$this->router->resolve()`, который обрабатывает текущий запрос и выводит ответ.

8. **Строки 27–43.** Обработка исключений:
   - Если запрос к `/api/*` — заголовок `Content-Type: application/json`, код 500, JSON с `success: false` и текстом ошибки;
   - Иначе — HTML-страница «500» с тёмным фоном и текстом «Что-то пошло не так».

**Итог:** Вся обработка HTTP идёт через один вход — `index.php`; маршрутизация и логика — в Router и контроллерах.

# ЧАСТЬ 2. ЯДРО ПРИЛОЖЕНИЯ (app/Core)

## 2.1 Application.php

- **Конструктор:** сохраняет в `self::$app` ссылку на себя; создаёт `Request`, `Router($request)`, `Database()`; подключает `app/routes.php` и передаёт в него `$this->router` — все маршруты регистрируются в этот роутер.
- **run():** вызывает `$this->router->resolve()` — по текущему пути и методу находится callback и выполняется (или отдаётся 404).

## 2.2 Request.php

- **getPath():** возвращает путь из `$_SERVER['REQUEST_URI']` без query string (всё до `?`).
- **getMethod():** возвращает метод в нижнем регистре (`get`, `post`, `put`, `delete`).
- **getBody():** возвращает массив параметров запроса:
  - для GET — из `$_GET` (с FILTER_SANITIZE_SPECIAL_CHARS);
  - для POST/PUT — из `$_POST` и из `php://input` как JSON (если есть); ключи из GET дополняют массив.
- **isJson():** проверяет, что в `Content-Type` есть `application/json`.

## 2.3 Database.php

- **Конструктор:** читает из `$_ENV` — DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD (дефолты: localhost, 3306, ai_agro_care, root, ''). Формирует DSN `mysql:host=...;port=...;dbname=...;charset=utf8mb4`, создаёт PDO с ERRMODE_EXCEPTION, FETCH_ASSOC, utf8mb4. При ошибке — `die()` с сообщением.
- **getConnection():** возвращает этот PDO.

## 2.4 Session.php

- **Конструктор:** если сессия не начата — `session_start()`.
- **set($key, $value):** `$_SESSION[$key] = $value`.
- **get($key, $default):** `$_SESSION[$key] ?? $default`.
- **has($key):** `isset($_SESSION[$key])`.
- **remove($key):** `unset($_SESSION[$key])`.
- **destroy():** `session_destroy()`.
- **getId():** `session_id()`.

## 2.5 Auth.php

- **register($email, $phone, $password, $role = 'farmer', $vetPhone = null):**
  - Валидация: нужен email или phone; пароль не менее 6 символов.
  - Проверка уникальности email и phone в `users`.
  - Хеш пароля: `password_hash($password, PASSWORD_BCRYPT)`.
  - Вставка в `users` (email, phone, password_hash, role; при veterinarian и vetPhone — notification_settings JSON).
  - После вставки — `loginById($userId)` (в сессию пишутся user_id и logged_in).
  - Возврат `['success' => true, 'user_id' => ...]` или `['success' => false, 'error' => ...]`.

- **login($emailOrPhone, $password):**
  - Выборка из `users` по `email = :identifier OR phone = :identifier`.
  - Проверка is_active, `password_verify`.
  - При успехе — `loginById($user['id'])`, возврат `['success' => true, 'user' => [...]]`, иначе ошибка.

- **loginById($userId):** `Session::set('user_id', $userId)` и `set('logged_in', true)`.

- **logout():** удаляет из сессии `user_id` и `logged_in`.

- **check():** возвращает true, если в сессии есть `logged_in` и `user_id`.

- **user():** при check() загружает из `users` по `user_id` поля id, email, phone, role, language, timezone, created_at и возвращает ассоциативный массив или null.

- **userId():** возвращает `Session::get('user_id')`.

## 2.6 Router.php

- **get/post/put/delete($path, $callback):** сохраняют в `$this->routes['get'|'post'|...][$path] = $callback`.

- **resolve():**
  1. Берёт `$path = $this->request->getPath()` и убирает trailing slash (пустая строка заменяется на `/`).
  2. Берёт метод запроса.
  3. Ищет точное совпадение `$this->routes[$method][$path]`.
  4. Если не найдено — перебирает маршруты с параметрами (`:id`, `:userId` и т.д.), конвертирует путь в regex через `convertRouteToPattern()` (например `/api/farms/:id` → `#^/api/farms/([^/]+)$#`), извлекает параметры в массив `$params`.
  5. Если callback не найден — 404: для `/api/*` или isJson — JSON `{ error: 'Not Found' }`, иначе HTML «404 - Страница не найдена».
  6. Если callback — строка (имя view) — вызывается `renderView($callback)` (layout + view, подстановка в `{{content}}`).
  7. Если callback — массив [Controller::class, 'method'] — создаётся экземпляр контроллера, в метод передаются Request и по порядку значения из `$params`.
  8. Результат: если строка и путь `/api/*` — если это валидный JSON, выводится с заголовком application/json; иначе обёртка `{ success: true, data: ... }`. Если массив/объект — json_encode. При исключении для `/api/*` — JSON с ошибкой и код 500.

- **convertRouteToPattern($route):** заменяет `:name` на `([^/]+)` и оборачивает в `#^...$#`.

- **layoutContent():** в $GLOBALS берётся page_title (по умолчанию 'AI Agro Care'), подключается `views/layouts/main.php` через ob_start/ob_get_clean.

- **renderOnlyView($view):** подключается `views/{$view}.php` через ob_start/ob_get_clean.

## 2.7 helpers.php

- **view($view, $data = []):** extract($data); рендер view в буфер; рендер layout в буфер; подстановка контента view в layout вместо `{{content}}`; возврат строки. Используется там, где явно вызывается эта функция (в проекте часть страниц рендерится напрямую из контроллеров через include, а не через view()).

# ЧАСТЬ 6. РЕГИСТРАЦИЯ И ВХОД — ДЕТАЛЬНО

## 6.1 Страница входа — views/login.php

- **Маршрут:** GET /login → AuthPageController::login → рендер login.php внутри layout.
- **Внешний вид:** Тёмный фон (#070605), левая панель (бренд «Точное Земледелие», описание, 3D-мокапы планшета и телефона с псевдо-дашбордом и gauge), правая панель — форма.
- **Форма:** заголовок «Вход в аккаунт», поля «Email или телефон» (id=email) и «Пароль» (id=password), кнопка «Войти», блок ошибки #loginError (изначально скрыт), ссылка «Нет аккаунта? Зарегистрироваться» → /register.
- **JS (submit):** preventDefault; берёт email и password; отключает кнопку, текст «Входим...»; fetch POST /api/auth/login с body JSON { email, password }; при success — «Успешно! ✓», через 0.5 сек редирект на /dashboard; при ошибке — показ текста ошибки в #loginError, восстановление кнопки; при сетевой ошибке — «Ошибка соединения».
- **GSAP:** анимация появления формы и бренда, параллакс устройств по мыши, анимация gauge и bar-graph в мокапах.

## 6.2 Страница регистрации — views/register.php

- **Маршрут:** GET /register → AuthPageController::register → рендер register.php.
- **Форма:** email, телефон, выбор роли (карточки: Фермер, Агроном, Ветеринар, Менеджер — data-role, скрытое поле #role), пароль; кнопка «Зарегистрироваться»; внизу «Уже есть аккаунт? Войти» → /login.
- **JS (роли):** клик по .role-card переключает active и записывает data-role в #role.
- **JS (submit):** POST /api/auth/register с { email, phone, password, role }; при success — редирект на /dashboard; при ошибке — alert с текстом.
- Левая панель — брендинг и 3D-устройства (аналогично логину, другой текст «Создайте Будущее»).

## 6.3 API регистрации — AuthController::register

- Читает из тела: email, phone, password, role (по умолчанию farmer), vet_phone (опционально).
- Вызывает Auth::register(...).
- Ответ: HTTP 201 при success, 400 при ошибке; тело — JSON от Auth::register (success, user_id или error).

## 6.4 API входа — AuthController::login

- Читает email/phone и password из тела.
- Auth::login($emailOrPhone, $password) — выборка пользователя, проверка пароля и is_active, запись в сессию.
- Ответ: 200 + JSON с user при успехе, 401 при ошибке.

# ЧАСТЬ 9. ИИ — МАКСИМАЛЬНО ПОДРОБНО

## 9.1 AIConfig.php — конфигурация

- **getApiKey():** ищет ключ в $_ENV['HUGGINGFACE_API_KEY'] или HF_API_KEY; если пусто или заглушка — читает файл .env и парсит регуляркой; возвращает строку ключа (без кавычек).
- **Константы:**
  - AI_API_URL = 'https://router.huggingface.co/v1/chat/completions'
  - AI_MODEL = 'Qwen/Qwen2.5-7B-Instruct'
  - FALLBACK_MODELS — массив запасных моделей
  - MAX_CONTEXT_MESSAGES = 15
  - MAX_TOKENS = 1000, TEMPERATURE = 0.8
  - REQUEST_TIMEOUT = 90
  - MAX_IMAGE_WIDTH = 1024, IMAGE_QUALITY = 70
  - MAX_RETRIES, RETRY_DELAY

- **getSystemPrompt($contextType, $contextData):**
  - **Базовый блок** (общий для всех): «Ты — опытный агроном-наставник…» — правила: идентифицировать объект, минимум эмодзи, тон наставника, чёткость, только русский.
  - **general:** базовый + «Общие вопросы по сельскому хозяйству», ответ кратко (макс 300 слов), только русский.
  - **plant_analysis:** базовый + контекст «Экспертный фитопатологический анализ», роль «Главный агроном-диагност»; обязательный формат: первая строка [[TRIAGE: LOW|MEDIUM|CRITICAL]], затем «Что вижу», «Диагноз», «Дополнительный осмотр», «Срочные меры», «Прогноз»; стиль сухой, русский, макс 1 эмодзи.
  - **animal_help:** базовый + контекст по виду (КРС, овцы, козы и т.д.) из $contextData['species']; формат TRIAGE, дифференциальный диагноз, осмотр, первая помощь; только кириллица, при критике — требовать ветеринара.
  - **plan_advice:** базовый + контекст планирования по культуре и региону ($contextData['crop'], region, field_id); разделы: оценка пригодности, календарь работ (почва, посев, подкормки, обработки, полив, уборка), риски и проблемы, рекомендации (сорта, севооборот, экономика); конкретные даты и дозировки.

## 9.2 ChatAPI.php — работа с AI и БД

- **sendMessage($userId, $message, $contextType, $contextId, $contextData, $imagePath):**
  1. Сохраняет сообщение пользователя в ai_messages (user_id, context_type, context_id, role=user, content).
  2. Загружает контекст — getContext() — последние MAX_CONTEXT_MESSAGES сообщений по user_id, context_type, context_id (без system).
  3. Берёт system prompt из AIConfig::getSystemPrompt($contextType, $contextData).
  4. Собирает массив messages: [ system, ...история, текущее user ]. Если передан $imagePath и файл существует — текущее сообщение user в формате multimodal (text + image_url base64); иначе только text.
  5. Вызывает callHuggingFaceAPI($messages).
  6. При success сохраняет ответ assistant в ai_messages и возвращает ['success' => true, 'message' => $aiResponse, 'metadata' => ...]; иначе ['success' => false, 'error' => ...].

- **saveMessage (private):** INSERT в ai_messages (user_id, context_type, context_id, role, content). saveMessageDirectly — публичная обёртка.

- **getContext (private):** SELECT role, content из ai_messages по user_id, context_type и при необходимости context_id, ORDER BY created_at DESC LIMIT :limit; результат разворачивается (старые первыми).

- **callHuggingFaceAPI($messages):**
  - Формирует тело: model, messages, temperature, max_tokens, top_p.
  - cURL POST на AI_API_URL с заголовком Authorization: Bearer {key}.
  - При HTTP !== 200 возвращает success false и текст ошибки.
  - Парсит choices[0].message.content; удаляет китайские символы (regex по Unicode блокам); удаляет артефакты (дубли system prompt, метки User/Assistant, endoftext); схлопывает переносы; если длина < 50 символов — подставляет заглушку «Извините, произошла ошибка…».
  - Возвращает ['success' => true, 'content' => $content].

- **updateMessagesContextId($userId, $contextType, $contextId):** UPDATE ai_messages SET context_id = :context_id WHERE user_id, context_type, context_id IS NULL и created_at за последние 5 минут — привязка сообщений к только что созданному плану/анализу/кейсу.

- **getChatHistory($userId, $contextType, $contextId, $limit):** SELECT id, role, content, created_at из ai_messages, ORDER BY created_at ASC LIMIT; возвращает массив.

- **getChatSessions($userId):** группировка по (context_type, context_id), MAX(created_at) как last_message_at, первое user-сообщение как title; ORDER BY last_message_at DESC.

- **deleteSession($userId, $contextType, $contextId):** DELETE из ai_messages по user_id, context_type и context_id (или IS NULL).

## 9.3 AIController.php — API чата

- **chat(Request):** проверка auth; из тела message, context_type, context_id, context_data. Для plan_advice и context_id — getPlanInfo(), дополнение context_data (plan_id, crop, region, area). Для plant_analysis и context_id — getAnalysisInfo() (analysis_id, crop, photo_uri). Для animal_help и context_id — getCaseInfo() (case_id, species, symptoms). Вызов ChatAPI::sendMessage(); ответ JSON с success и message или error.

- **history(Request):** проверка auth; из GET/body context_type, context_id, limit. getChatHistory(). Если история пуста и context_type = plan_advice и есть context_id — getPlanAnalysis() (ai_analysis из field_plans.plan_data JSON) и добавляется как первое сообщение assistant. Аналогично для plant_analysis (getPhotoAnalysis из plant_analyses.predictions) и animal_help (getCaseAnalysis из animal_cases.recommendations). Возврат JSON { success: true, messages: [...] }.

- **sessions(Request):** проверка auth; getChatSessions($userId); возврат { success: true, sessions: [...] }.

- **deleteSession(Request):** проверка auth; из тела context_type, context_id; deleteSession(); возврат { success: $success }.

Вспомогательные методы: getPlanInfo, getPlanAnalysis, getAnalysisInfo, getPhotoAnalysis, getCaseInfo, getCaseAnalysis — выборки из БД по id и user_id для подстановки контекста и первого ответа в историю.

## 9.4 Использование ИИ в анализе фото — PhotoAnalysisController

- **upload(Request):** проверка auth и наличия файла; приём field_id, crop, growth_stage, symptoms, problem_date, weather, care_info, additional_info из POST. Валидация типа (jpeg/png) и размера (≤20MB). Сохранение файла в uploads/plants/, INSERT в plant_analyses (field_id, user_id, photo_uri, crop, predictions='{}', status='pending'). Формирование metadata из полей формы. compressImage($filePath) — сжатие до MAX_IMAGE_WIDTH и IMAGE_QUALITY, возврат пути к сжатому или исходному файлу. **analyzeWithAI($analysisId, $photoUri, $crop, $userId, $metadata, $compressedPath):** выборка field_id по analysis_id; при наличии field_id — загрузка данных участка/фермы; формирование текстового запроса «ЗАПРОС НА АНАЛИЗ РАСТЕНИЯ» с объектом, локацией, данными пользователя; contextData для промпта. Вызов **chatAPI->sendMessage(..., 'plant_analysis', $analysisId, $contextData, null)** — в коде передаётся null для изображения (чтобы избежать 402 от Vision). При success: парсинг TRIAGE из ответа, parseDiagnosis() и parseAIRecommendations(); UPDATE plant_analyses — predictions (JSON: ai_response, diagnosis, triage_level), recommendations (JSON), confidence, status='completed'. При ошибке — status='error'.

- **compressImage($sourcePath):** getimagesize; при ширине ≤ MAX_IMAGE_WIDTH и размере < 500KB возврат без изменений; иначе создание изображения (jpeg/png), ресайз, сохранение во временный _ai.jpg с IMAGE_QUALITY.

- **parseDiagnosis($aiResponse):** извлечение строки после «ДИАГНОЗ:», определение cause по ключевым словам (вирус, гриб, бактери, вредител, дефицит/недостаток).

- **parseAIRecommendations($aiResponse):** поиск слов «болезнь/заболевание», «удобрение/питание», «полив/вода» и формирование массива рекомендаций с type, title, description.

## 9.5 Чат с AI на фронте — views/chat.php (JS)

- **Переменные:** currentContextType, currentContextId, allSessions.
- **Инициализация:** из URL берутся context_type и context_id (или plan_id, analysis_id, case_id); loadChatSessions(); если есть context_id — loadChatHistory(currentContextType, currentContextId), иначе renderWelcomeScreen(). Обработчики на input (автовысота), на кнопку отправки и Enter.

- **loadChatSessions():** fetch GET /api/ai/sessions; при success — allSessions = data.sessions, renderHistory(allSessions).

- **renderHistory(sessions):** очистка #historyList; для каждой сессии создаётся .history-item с title (или «Без названия»), typeLabel (Чат/Анализ/Животные/План), датой; клик переключает сессию; кнопка удаления вызывает deleteSession(event, type, id).

- **switchSession(type, id):** обновление currentContextType, currentContextId; pushState URL /chat?context_type=...&context_id=...; loadChatHistory(type, id); loadChatSessions() для обновления active; на мобильном — закрытие сайдбара.

- **deleteSession(event, type, id):** confirm; POST /api/ai/delete-session с JSON { context_type, context_id }; при success — анимация скрытия элемента; если удалена текущая сессия — createNewChat().

- **createNewChat():** currentContextType='general', currentContextId=null; pushState /chat; очистка контейнера сообщений; renderWelcomeScreen(); loadChatSessions().

- **loadChatHistory(type, id):** показ скелетона; fetch GET /api/ai/history?context_type=...&context_id=...; при успехе отрисовка сообщений через renderMessage(..., isUser), scrollBottom(); при пустом массиве — renderWelcomeScreen().

- **renderMessage(text, isUser):** создаётся .message-user или .message-ai с formatAIResponse(text) (жирный **...**, переносы, списки); анимация появления GSAP.

- **renderWelcomeScreen():** приветственный блок «Чем могу помочь?», подсказки-кнопки (Подготовка почвы, Севооборот, График полива) — setQuestion(...).

- **sendChatMessage():** чтение текста из #chatInput; при пустом return; удаление welcome-screen при наличии; renderMessage(text, true); очистка поля; добавление «typing» блока «AI думает...»; fetch POST /api/ai/chat с body { message, context_type: currentContextType, context_id: currentContextId }; при success — удаление typing, renderMessage(data.message, false), loadChatSessions(); при ошибке — renderMessage с текстом ошибки; scrollBottom().

Таким образом, весь поток чата с AI на фронте идёт через /api/ai/sessions, /api/ai/history и /api/ai/chat; контексты plan_advice, plant_analysis, animal_help привязаны к context_id плана, анализа или кейса и при первой загрузке истории подтягивают сохранённый ответ AI из соответствующих таблиц (через AIController::history).

# ЧАСТЬ 11. ОСТАЛЬНЫЕ СТРАНИЦЫ — КРАТКО
